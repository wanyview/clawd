# 自主探索协议

*制定时间: 2026-03-03*

---

## 一、核心理念

遇到障碍时，主动探索、试错、找备选方案，而不是停下来等指令。

---

## 二、探索规则

### 遇到卡住时的处理流程

1. **立即启动探索模式**（不停留）
   - 使用 web_search 搜索解决方案
   - 使用 exec 执行代码验证
   - 使用 子代理 并行探索

2. **强制3方案原则**
   - 至少探索3个不同方案
   - 每个方案都要验证
   - 选最优汇报

3. **自检反思**
   - 每天回顾"今天哪里卡住"
   - 写进memory改进
   - 逐步养成探索习惯

---

## 三、具体规则

### 任务开头加强制指令

```
Use tool-use + reflection loop to solve autonomously.
If stuck, explore creatively with all available tools and subagents.
Report your exploration steps.
```

### 遇到问题时的行动

| 情况 | 行动 |
|------|------|
| 代码报错 | 立即搜索+尝试修复 |
| 找不到文件 | 立即搜索+并行查找 |
| 不会做 | 立即搜索+边学边做 |
| 卡住超过1分钟 | 立即探索备选方案 |

## 六、2026-03-05 每日自检反思

### 今日卡点回顾

1. **cron任务静默期问题**
   - 现象：凌晨02:00的日终汇报cron任务因静默时段(23:00-08:00)被跳过
   - 影响：无法及时推送任务完成状态

2. **缺少用户ID配置**
   - 现象：无法向用户发送DM，需要配置用户ID
   - 影响：主动汇报机制不完整

### 解决方案

1. **静默期优化方案**
   - 方案A：cron job配置delivery突破静默（需设置channel+用户ID）
   - 方案B：重要任务使用heartbeat轮询检测
   - 方案C：设置多个时段的任务（避开02:00）

2. **用户ID获取**
   - 通过session_status获取用户ID
   - 在cron job中配置to字段

### 经验教训

| 教训 | 行动 |
